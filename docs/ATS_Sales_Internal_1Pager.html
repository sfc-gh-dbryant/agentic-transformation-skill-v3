The Critical Distinction: Authoring vs. Execution

ATS authors the pipeline. It is not the pipeline.

When a customer asks "will ATS automatically transform new Bronze data every night?" — the answer is no, and it shouldn't. ATS answers the hardest question: given raw tables I've never seen before, what should my Silver layer look like? Once answered, running the LLM nightly to re-make decisions that are already made is wasteful. The correct path:

Think of ATS like an architect's AI-assisted design software — it accelerates how fast blueprints are produced. You don't leave the design software running in the building after construction.

Brownfield Safeguards — Existing Data Is Protected

ATS runs safely in environments that already have Silver tables, partial pipelines, or existing schema objects. Three built-in controls prevent accidental overwrites — all default to the safe position.

🔒 Dry Run
Default: ON
Generates and logs all DDL but never executes. Customer reviews every statement first. Must be explicitly set to OFF to run anything.
🔒 No Overwrite
Default: OFF
If a target table already has rows, the Executor skips that table, logs TARGET_EXISTS_NO_OVERWRITE, and continues with the remaining tables. Other tables in the same run are unaffected.
🔀 Brownfield Mode
Default: OFF
When enabled, skips existing Silver tables entirely (logs as EXISTING, not failures) and only processes gaps. Use when some tables are already built.
🔑
Two-Key Interlock
Enabling overwrite requires both dry_run=FALSE AND overwrite_existing=TRUE set simultaneously. Setting one without the other returns an error.
⚠️
Limitation: Existing Pipeline Coexistence
The row-count check only protects populated tables. It does not protect empty tables (e.g. a new dbt model not yet run), Dynamic Tables, or Views — CREATE OR REPLACE TABLE will silently replace them. Use output_schema isolation: configure ATS to write to a dedicated staging schema (e.g. SILVER_PROPOSED) separate from any schema your existing pipelines own.

Handling Common Sales Conversations

"Can we include ATS as a line item in the SOW?"
No. ATS is how PS delivers — not what PS delivers. Line items are the pipeline outputs. Reference the methodology in the SOW narrative but do not scope it as a deliverable.
"The customer wants to see a demo."
Yes — the Streamlit app is designed for this. Demo the pipeline running, the agents working, the observability. Frame it as "this is how we build pipelines" — not "this is what you're buying."
"The customer wants to own ATS post-engagement."
Fully supported — great upsell to an enablement engagement. The framework deploys in their account and they can extend it. But: their team needs DE expertise to operate it, and Snowflake PS support ends with the engagement.
"Can we productize ATS for the Marketplace?"
Not at this time. ATS is an internal PS accelerator. Any Marketplace or productization conversation must involve PS leadership before any customer discussion.
